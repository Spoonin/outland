## D-001 — Артефакт = процессный навык `project-context`   (2026-06-13, статус: accepted)
**Контекст:** нужно решить, что именно мы строим в первую очередь.
**Решение:** строим процессный навык — свод правил, по которым Claude ведёт живой контекст проекта. Outland — первый проект-применение и тестовый пример.
**Альтернативы:** методичка по геймдизайну (отклонено — не переиспользуема); статичная обёртка вокруг одной игры (отклонено — не масштабируется на другие проекты).

## D-002 — Контекст — живой документ между сессиями   (2026-06-13, статус: accepted)
**Контекст:** боль — каждый новый чат начинается с нуля, проект приходится пересказывать.
**Решение:** навык отвечает и за создание, и за поддержание актуальности контекста между сессиями. Ценность — «передача эстафеты».
**Альтернативы:** разовая генерация документа без сопровождения (отклонено — быстро устаревает).

## D-003 — Источник правды — файл в git-репо   (2026-06-13, статус: accepted)
**Контекст:** где хранить канонический контекст.
**Решение:** локальный файл в git-репозитории: `documents/index.md` (корневое оглавление) + иерархия под-документов рядом.
**Альтернативы:** chat memory / память модели (отклонено — не персистентно, не под контролем пользователя).

## D-004 — Основная среда — Claude Code   (2026-06-13, статус: accepted)
**Контекст:** в какой среде навык работает штатно.

**Альтернативы:** только обычный чат (отклонено — нет доступа к диску, всё вручную).

## D-005 — Структура иерархии под-документов   (2026-06-13, статус: accepted)
**Контекст:** какой набор файлов держать рядом с `index.md` (Узел 5).
**Решение:** 5 файлов — `index.md` (карта+питч+статус+своды), `decisions.md` (этот журнал), `open-questions.md`, `mechanics.md`, `references.md`. Под-документы создаются **лениво** — файл появляется на диске только при наличии реального содержимого; до этого помечен «не создан» в карте `index.md`. Глоссарий — секцией внутри `mechanics.md`, отдельный `glossary.md` заводим только при разрастании (см. будущий узел бюджета объёма).
**Альтернативы:** сразу заводить пустые плейсхолдеры (отклонено — разрастание пустышек); отдельный `glossary.md` с самого начала (отложено до необходимости).

## D-007 — Состав секций `index.md`   (2026-06-13, статус: accepted)
**Контекст:** какой финальный набор секций держать в корневом файле (Узел 6).
**Решение:** 6 основных секций — Питч/тезис, Статус, Карта документов, Свод ключевых решений, Топ открытых вопросов, Лог изменений — плюс 2 дайджест-секции (Ключевые механики, Референсы), где допускается строго одна строка на пункт со ссылкой на под-документ. Дайджесты оставлены ради самодостаточности входа (D-002, «передача эстафеты»): новая сессия понимает суть проекта, прочитав только `index.md`.
**Альтернативы:** чисто навигационный index без механик/референсов (отклонено — заставляет открывать под-документы только чтобы понять, о чём проект); полные секции механик/референсов в index (отклонено — дублирование с под-документами, гарантированный рассинхрон, разрастание против бюджета объёма).

## D-008 — Триггеры навыка   (2026-06-13, статус: accepted)
**Контекст:** при каких сигналах срабатывает навык; что писать в `description` SKILL.md (Узел 7).
**Решение:** три класса триггеров — (1) контекстный/автоматический: репо с `documents/index.md` → читать первым; (2) явные фразы («обнови контекст», «зафиксируй решение», «продолжаем проект», «подхвати где остановились», «заверши сессию»); (3) границы сессии (старт — прочитать+свериться, конец — синхронизировать+предложить коммит). Плюс режим создания с нуля по явной фразе; само отсутствие `index.md` навык НЕ активирует. Заодно починен фронтматтер SKILL.md (нормальный YAML `name`/`description`), тело подтянуто под D-005/D-007.
**Альтернативы:** активировать навык в любом репо без `index.md` (отклонено — навык лез бы в каждый проект); только явные фразы без авто-контекста (отклонено — теряется главная ценность «передачи эстафеты»).

## D-009 — Ядро игры = граф зависимостей на «спрос ≥ MES»   (2026-06-13, статус: accepted)
**Контекст:** как превратить тезис о незамыкаемых циклах в работающую механику, а не лозунг (детали → `mechanics.md` §1).
**Решение:** ядро — граф зависимостей, где у каждого узла есть минимальный эффективный масштаб (MES) и спрос колонии D. Узел эффективно локализуем при `D ≥ MES` + локализованном поддереве. Спрос агрегируется вверх по дереву (общий апстрим кормится суммой потребителей), но ограничен населением → положительная обратная связь упирается в потолок → асимптота замыкания, последний % недостижим в принципе.
**Альтернативы:** чистый tech-tree без экономики масштаба (отклонено — не доказывает тезис, автономия выглядела бы достижимой); скриптовый «проигрыш по сюжету» (отклонено — игрок должен сам упереться в стену, иначе тезис не прочувствуется).

## D-010 — Темп прозрения: соблазн ложной метрикой   (2026-06-13, статус: accepted)
**Контекст:** ключевое балансное требование — игрок должен осознать схождение к провалу постепенно, не моментально и не внезапным обрывом (детали → `mechanics.md` §2).
**Решение:** трёхактная кривая (соблазн → сомнение → прозрение), эмерджентная из чисел. Игрока соблазняет ярко показанная автономия %, тогда как настоящие метрики (пол импорта F, стоимость закрытия следующего %) растут тускло и с задержкой. Балансные ручки: экспонента MES по тирам (k), скорость износа → наклон F, бюджет массы окна B vs рост спроса, глубина/число чёрных узлов, график раскрытия информации. Избегаем двух ошибок: «слишком очевидно» (рельсы) и «слишком скрыто» (нечестный обрыв).
**Альтернативы:** мгновенно честная подача всех метрик (отклонено — нет игры, нет соблазна); скриптовые акты по таймеру (отклонено — должно следовать из чисел, иначе ощущается как рельсы).

## D-011 — Импорт-эскейп + раскрытие сверху вниз   (2026-06-13, статус: accepted)
**Контекст:** как сделать раскрытие дерева постепенным, если производство кажется обязанным идти снизу вверх (детали → `mechanics.md` §1.2, §3). Слоган: «либо импортируешь компонент — либо осознаёшь всю глубину своей проблемы».
**Решение:** дерево детерминировано (на нём держится баланс), недетерминировано только знание игрока (туман) + события + оценки. Импорт — всегда доступный вход на каждом узле, поэтому локализация требует лишь доступных ПРЯМЫХ входов (локально ИЛИ импортом); строить снизу вверх обязателен только полностью локализуемый кусок. Спрос разворачивается сверху вниз тир за тиром по решениям make-or-buy — это и есть раскрытие тумана. Чёрный узел вскрывается поздно, т.к. до погони за последними % не было причины копать так глубоко. Оценки скрытых узлов смещены в оптимизм и уточняются разведкой — это склеивает недетерминированность с темпом прозрения (D-010).
**Альтернативы:** процедурно-случайная структура дерева (отклонено — нельзя гарантировать схождение к провалу и баланс §2); строго снизу вверх без импорт-эскейпа (отклонено — делает постепенное раскрытие логически невозможным).

## D-012 — Двигатель ходов = синодические окна + масса-валюта   (2026-06-13, статус: accepted)
**Контекст:** что превращает статичный граф в игру во времени и откуда берётся бюджет массы B (детали → `mechanics.md` §4).
**Решение:** ход = одно синодическое окно (~26 мес, партия ~25–40 окон). Дефицитная валюта — масса на грунт Марса (тирания Циолковского: уравнение ракеты). Каждое окно игрок делит бюджет B между обязательным импорт-floor F, капиталом (автономия) и людьми; F забирает B первым → по мере F→B свободный капитал→0 (механически Акт 3 из §2). Лаг 1–3 хода усиливает ловушку оптимизма (D-010/D-011). Леверы массы (ISRU/локальное топливо) — узлы графа → двигатель и граф сцеплены. Крупнозернистое время + ратчет населения → хрупкость.
**Альтернативы:** непрерывное/реальное время (отклонено — синодические окна дают крупный масштаб и медленное прозрение, физически честны); масса как абстрактные «кредиты» без уравнения ракеты (отклонено — теряется тирания Циолковского, ключевой источник дороговизны).

## D-013 — Износ как двигатель роста пола импорта   (2026-06-13, статус: accepted)
**Контекст:** что порождает и растит пол импорта F и держит узлы привязанными к импорту (детали → `mechanics.md` §5).
**Решение:** два рода износа — утечка вещества в замкнутых циклах (КПД рециклинга η<100% → вечная подпитка) и деградация капитала (MTBF → запчасти → замена). Каждый локализованный узел несёт хвост обслуживания (запчасти+подпитка); локализация снижает основной импорт, но добавляет меньший импорт обслуживания — ни одна ветка не обнуляется. Хвост рекурсирует вниз как производство → погоня за нулём упёка загоняет в чёрные узлы. Износ копится по окнам → F растёт с задержкой (топливо для прозрения, D-010). η-асимптота = механика «асимптоты замыкания». Отложенное обслуживание → каскадные отказы (хрупкость).
**Альтернативы:** локализованные узлы бесплатны навсегда (отклонено — F не растёт, нет зажима, нет тезиса); износ как чисто случайные поломки без хвоста обслуживания (отклонено — теряется связь «локализация оставляет импортный остаток» и рекурсия глубины).

## D-014 — MES скрыт, выводится игроком эмпирически   (2026-06-13, статус: accepted)
**Контекст:** показывать ли игроку MES числом (детали → `mechanics.md` §1.3, §1.6).
**Решение:** MES детерминирован в модели (баланс держится, D-011), но игроку НЕ показывается. Игрок наблюдает только реализованную удельную цену после постройки и выводит порог эмпирически — пробуя масштабы и агрегируя спрос (как сэмплируя скрытую функцию). Статусы 🟢/🟡 обнаруживаемые, а не подписанные: эффективен узел или субмасштабен, выясняется по эксплуатации. До-постройочная оценка оптимистично смещена (§3.4) → субмасштабный завод иногда яма, в которую масса уже вложена (лаг §4.4).
**Альтернативы:** показывать MES числом (отклонено — игрок считает выгоду до постройки, рушится соблазн D-010 и туман D-011); полностью случайный MES (отклонено — ломает баланс; скрытость ≠ случайность).

## D-015 — Хрупкость популяции: у навыков тоже есть MES   (2026-06-13, статус: accepted)
**Контекст:** как смоделировать малое население как источник хрупкости, не как отдельную подсистему (детали → `mechanics.md` §6).
**Решение:** население двойственно — источник спроса D (растит F) и источник труда/навыков (узлы надо эксплуатировать). Узлы требуют специалистов, не только капитал. Ключевое: у навыков тоже есть MES — локальное обучение специальности требует учебной базы с минимальным масштабом; колония ниже порога импортирует экспертизу навсегда (человеческий пол импорта, структурно тот же F). Хрупкость = единые точки отказа (соблазн держать 1 специалиста без резерва); потеря гасит узел → каскад (§5.6), знание теряется безвозвратно, если не передано. Тезис незамыкаемости распространён на кадры.
**Альтернативы:** население как простой множитель труда без специальностей (отклонено — теряется хрупкость и «человеческий пол импорта»); генетическая минимальная популяция как главный фокус (отклонено — игра про замыкание циклов, а не про демографию; порог навыков важнее).

## D-016 — Жанр и game loop   (2026-06-13, статус: accepted)
**Контекст:** что игрок физически делает каждый ход; как модель становится игрой (детали → `mechanics.md` §7).
**Решение:** пошаговая экономическая стратегия / менеджмент колонии, базовый глагол make-or-buy по скрытому графу под бюджетом массы; платформа — компьютер. Петля окна: обзор → разведка/план → распределение B → коммит → ход времени (~2,2 года) с запаздывающими последствиями. Ритм «планируй в неопределённости → коммить массу → жди → узнавай последствия». Поверхностная цель — автономия→100%, реально недостижимая (соблазн).
**Альтернативы:** реальное время / Factorio-подобный автоматизатор (отклонено — теряется крупный масштаб окон и честный расчёт); PvP/конкуренция (отклонено — противник игры — физика, не оппонент).

## D-017 — Победа переосмыслена: трагедия, а не триумф   (2026-06-13, статус: accepted)
**Контекст:** что значит «выиграть» в игре, доказывающей невозможность автономии (детали → `mechanics.md` §7.4–7.6).
**Решение:** триумфальной победы-автономии нет. Концовки: коллапс / отмена проекта Землёй (импорт неподъёмен) / асимптотический штиль (выживание при вечной зависимости). Лучший исход — стабильная, но вечно зависимая колония, что само доказывает тезис. Финальный дебриф показывает кривую в ретроспективе (плато автономии, F, чёрные узлы) — «заслуженное прозрение» (§2.5). Реиграбельность подтверждает тезис: разные старты → стена в разных местах.
**Альтернативы:** классическая победа при 100% автономии (отклонено — прямо противоречит тезису); чистый невыигрываемый софтлок без дебрифа (отклонено — игрок почувствует обман, а не прозрение, ср. §2.5).

## D-018 — UI: хаб-дашборд + спицы, граф как drill-down   (2026-06-13, статус: accepted)
**Контекст:** как игрок видит и трогает модель; представление «доски»-графа (детали → `mechanics.md` §8).
**Решение:** архитектура хаб-дашборд + спицы. Дашборд (дом) — ресурсы, люди/специалисты, грузы и люди в пути, состояния инфраструктуры/объектов, громкая автономия %, тусклые F/счёт импорта. Граф НЕ постоянный фон, а контекстный drill-down «дерево объекта» (отступ-BOM) при добавлении/локализации; отдельная спица «панель импорта» при решении buy; экран «манифест окна» для распределения B. Дерево показывает make-or-buy, оптимистичную оценку с вилкой (MES скрыт, D-014), специалистов/bus-factor (D-015), туман с [разведать]. Подача: честное недоговаривание (не god-mode), асимметрия заметности (хорошее громко / плохое тихо, D-010), drill-down требует усилия → не-заглядывание готовит прозрение.
**Альтернативы:** постоянный полноэкранный граф как главный вид (отклонено — давит, плохо масштабируется на сотни узлов); чистый node-link граф вместо BOM (отклонено — хуже масштаб; фокус-граф оставлен как опция для агрегации, §8.8).
